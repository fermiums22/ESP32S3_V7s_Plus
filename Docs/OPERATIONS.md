# Рабочие контуры робота

Этот файл — короткая операционная карта для следующих сессий. У робота три
разные прошивки. Их нельзя собирать или обновлять одной и той же командой.

## Архитектура

```text
Home Assistant
      |
      v
ESP32-S3 v7s-plus (ESPHome, главный контроллер и Modbus RTU master)
      |  Modbus RTU, адрес 1             |  Modbus RTU, адрес 2
      v                                  v
STM32F071                          ESP32-WROOM audio bridge
моторы, зарядка,                  GoPro BLE, RTMP-команды,
ИК и датчики                      I2S -> Bluetooth A2DP -> JBL
                                             |
                                             v
                                      GoPro HERO12
                                      RTMP -> HA/go2rtc
```

Windows не является частью рабочего контура. Он нужен только как компьютер
разработчика. Команды GoPro отправляет ESP32-WROOM, а S3 передаёт ему команды
через Modbus RTU с адресом `2`. Никакой постоянно запущенный Windows bridge не
нужен.

## Три разных пути обновления

### 1. STM32F071: временный BIN через уже работающую S3

Исходники STM32 находятся в `D:\w_space\V7s_Plus`. При изменении выводов,
таймеров или периферии сначала правится `V7s_Plus.ioc`, затем генерируется код
CubeMX и собирается Debug.

Из этого репозитория нормальный путь начинается командой:

```bat
build_stm32.bat
```

Она создаёт свежий `..\V7s_Plus\Debug\V7s_Plus.bin` из только что собранного
ELF. Затем этот BIN один раз загружается во временное хранилище уже работающей
S3:

```powershell
.\tools\upload_stm32_firmware.ps1 `
  -TargetHost v7s-plus.local `
  -Token '<stm32_upload_token>' `
  -Username '<web_server_username>' `
  -Password '<web_server_password>'
```

Значения берутся из локального `secrets.yaml` и не коммитятся. После статуса
`staged image ready` в Home Assistant нажимается штатная кнопка
`Flash staged STM32 firmware`.

S3 проверяет размер, CRC32 и векторную таблицу STM32F0. После попытки прошивки
временный файл удаляется. Постоянного STM32 BIN на S3, двух слотов и fallback-
образа нет.

Для изменения только STM32 запрещено запускать `compile.bat` или OTA S3. Это
прошивает не тот контроллер и создаёт лишний риск потерять связь с роботом.
Полная инструкция: [STM32_UPDATE.md](STM32_UPDATE.md).

### 2. ESP32-WROOM: отдельная OTA audio bridge

Проект WROOM находится в `D:\w_space\ESP32_V7s_Plus`. Собирать и обновлять его
нужно только из этого проекта его собственными `compile.bat` и OTA-командой.
Перед OTA на S3 открывается штатное окно обновления audio bridge: S3 временно
приостанавливает опрос Modbus адреса `2`, а WROOM включает Wi-Fi. После
обновления нужно дождаться перезапуска WROOM и закрыть окно OTA либо дождаться
его автоматического тайм-аута.

Это обновление меняет GoPro BLE, RTMP-управление, приём I2S и JBL A2DP. Оно не
перепрошивает S3 и не имеет отношения к временному STM32 BIN.

### 3. ESP32-S3: отдельная OTA главного контроллера

Только если изменены `v7s-plus.yaml`, компоненты ESPHome или код самой S3:

```bat
compile.bat
flash.bat v7s-plus.local
```

Этот путь обновляет только ESP32-S3. Он не собирает STM32 и не обновляет
ESP32-WROOM. OTA S3 не следует повторять ради добавления каждой отдельной
сущности: сначала готовится полный набор изменений, затем выполняется одна
сборка и одна прошивка.

## GoPro: видео, управление и звук

```text
HA native controls
      -> ESP32-S3
      -> Modbus RTU address 2
      -> ESP32-WROOM
      -> GoPro BLE: wake/sleep, Wi-Fi, start/stop RTMP, battery/state

GoPro HERO12
      -> RTMP over home Wi-Fi
      -> Home Assistant go2rtc stream gopro_robot
      -> camera entity/dashboard (video and camera audio)
```

Домашняя сеть GoPro выбирается и передаётся ей командой WROOM. SSID, пароль,
BLE-адрес и другие секреты хранятся только в локальных `secrets.yaml`, а не в
Git. Сущности управления камерой в HA должны быть нативными сущностями S3;
REST-команды к Windows и временные тестовые кнопки использовать нельзя.

При первичном BLE-сопряжении HERO12 может потребоваться открыть на камере
`Connections -> Pair/Connect`. После успешного сопряжения WROOM сам исполняет
команды, поэтому Windows исключён из эксплуатации.

RTMP идёт с GoPro напрямую в HA/go2rtc, а не через S3 или WROOM. Переключение
Wi-Fi может кратко оборвать поток; после подключения камеры к домашней сети он
должен снова появиться под именем `gopro_robot`. Наличие карточки без живого
изображения не считается рабочим видеопотоком.

Аудиодорожка GoPro приходит вместе с RTMP. Контур распознавания речи в HA:

```text
RTMP audio -> HA audio consumer -> Assist/Whisper -> текст/событие/команда HA
```

Не считать распознавание готовым, пока фактически не подтверждены аудиодорожка
живого RTMP и полученный Whisper текст.

## Зарядка, ИК и автономность STM32

Реальный выход управления DC/DC — `PC7 / TIM3_CH2`, PWM 20 кГц, активный
высокий уровень. Он задаётся в `V7s_Plus.ioc` как `CHARGER_ENABLE_PWM`.
Автономная логика зарядки и заезда на базу живёт в STM32 и не должна зависеть
от Home Assistant, Wi-Fi или доступности S3. HA даёт телеметрию и ручные
контролы для отладки.

Ток батареи в Modbus — знаковое значение в миллиамперах: отрицательное означает
заряд, положительное — разряд. ИК-приёмники публикуют только проверенные команды
и отфильтрованные состояния; сырой тестовый сниффинг не является штатной
телеметрией.

## Перед коммитом

1. Проверить `git status --short`: никаких паролей, токенов, BIN, дампов и
   временных тестовых файлов.
2. Выполнить `git diff --check`.
3. Собрать только ту прошивку, исходники которой менялись.
4. Коммитить исходники и документацию, но не generated output.
